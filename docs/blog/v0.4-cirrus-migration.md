# Draft: Five lines to move a Cirrus VM job to cove

Status: draft for v0.4 launch copy.

Cirrus Labs announced that Cirrus CI will shut down on Monday, June 1, 2026:
<https://cirruslabs.org/>. If your release process depends on `.cirrus.yml`,
hosted macOS tasks, or a Tart-backed CI queue, the deadline is not abstract.
Those jobs need a new scheduler and a new VM execution boundary before the
shutdown.

Cove is not a hosted CI service. That is the point. It is a local Apple Silicon
VM runner substrate for teams that want to own the host, the image, the fork,
the logs, and the teardown. You can keep GitHub Actions, Buildkite, or another
scheduler and use cove for the part Cirrus made valuable: fast macOS and Linux
VM jobs with a repeatable image boundary.

The migration model is simple. Build a runner image once, verify it before
jobs use it, then run every job in a fresh fork of that image:

```bash
cove action doctor
cove image verify --strict --newer-than 168h macos-runner:latest
cove action prepare-image macos-runner:latest --ttl 24h
go run ./cmd/cove-action -image macos-runner:latest -command 'xcodebuild test'
cove run -fork-from macos-runner:latest -ephemeral -headless
```

The fourth line is the action wrapper path. In GitHub Actions that becomes:

```yaml
- uses: ./.github/actions/cove-action
  with:
    image: macos-runner:latest
    command: xcodebuild test -scheme App -destination 'platform=macOS'
```

The fifth line is the underlying primitive. Cove materializes a disposable VM
from `macos-runner:latest`, waits for the guest agent, runs the command, emits
artifacts under `~/.vz/runs/<run-id>/`, then tears the fork down.

This differs from a warm persistent worker on purpose. Cove's soft-reset matrix
found that user deletion and warm cleanup do not form a reliable isolation
boundary for privacy-critical jobs. System Keychain entries, global
preferences, LaunchDaemons, TCC limits, and account lifecycle questions make a
"reset the same guest" story too easy to oversell. Cove's answer is more
boring and more defensible: fork from a known image and discard the child.

For teams that depended on Cirrus cache behavior, v0.4's cove action cache is a
whole-VM local cache. A cache hit changes which local image the job forks from;
it does not turn jobs into shared mutable VMs. Put lockfile hashes and tool
versions into the cache key, then let each job keep the same disposable-fork
boundary.

There are tradeoffs. Cove does not replace Cirrus's hosted queue. You need a
Mac host, a scheduler, a runner image, and operational ownership of the local
image store. In exchange, you get a private, inspectable path where images,
secrets, metrics, logs, and teardown all live on infrastructure you control.
