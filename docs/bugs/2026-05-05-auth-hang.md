# AuthorizationCreate hang during headed provisioning

Date: 2026-05-05

## Report

The headed fresh-VM path:

```bash
cove up -vm mlxgo-fresh-headed2-20260505 -user mlxqa -password mlxqa123 -gui -memory 8 -cpu 4 -no-shutdown
```

completed install, printed:

```text
10 file(s) need root privileges.
```

and then appeared to hang in Security.framework authorization.

The install stop path also printed a benign warning:

```text
warning: vm stop: domain=VZErrorDomain code=4: Invalid virtual machine state transition. Transition from state "stopped" to state "stopping" is invalid.
```

## Authorization call-site audit

Security.framework symbols are registered in `authorization.go`:

```text
AuthorizationCreate -> authCreate
AuthorizationExecuteWithPrivileges -> authExecute
```

Every repo call-site routes through the same native elevation path:

```text
PreWarm
  preWarmAuthorization
    createAuthorization

runElevated
  runElevatedManifestNative
    createAuthorization
    authExecute
```

There is no second direct `AuthorizationCreate` or
`AuthorizationExecuteWithPrivileges` caller.

## Gap

`createAuthorization` already runs `AuthorizationCreate` in a goroutine and
polls it with the no-UI and prompt-visible watchdogs. The next Security call,
`AuthorizationExecuteWithPrivileges`, was still synchronous in
`runElevatedManifestNative`.

That leaves the elevated-write step vulnerable to the same kind of authd or
SecurityAgent wedge after `AuthorizationCreate` returns or while the native
execution path is trying to launch the privileged child. The caller sees the
last provisioning message, `10 file(s) need root privileges.`, and then waits
inside native Security.framework code with no cove-level timeout.

The headed path also must not invoke native authorization on the registered
AppKit UI thread. Even a watchdogged authorization call should run from a
worker goroutine so the event loop can keep servicing SecurityAgent and cove UI
work.

## Fix

Keep one elevation surface:

```text
runElevated -> runElevatedManifestNative -> typed __elevated-op manifest
```

and harden that path:

- refuse native authorization from the registered UI thread;
- wrap `AuthorizationExecuteWithPrivileges` in the same watchdog style as
  `AuthorizationCreate`;
- keep the existing 90 second no-UI budget and 15 minute prompt-visible budget;
- keep the restricted/noninteractive fallback unchanged.

The stop warning is benign when the VM has already reached `stopped` between
the stop preflight and the completion handler. The installer should suppress
only that specific `VZErrorDomain` code 4 stopped-to-stopping race.
